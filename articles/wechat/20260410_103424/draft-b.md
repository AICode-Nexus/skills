# Draft B - Analytical Practical

## 标题方向

不会前端，也能一键起一个专业 React 项目

今天很多团队已经接受了一件事：产品、设计、运营、销售，甚至创始人自己，都可以直接和 AI 一起做原型、做 demo、做 MVP。

但真正把这件事做顺的人会很快发现，**最大的阻力并不在“让 AI 写页面”，而在“让 AI 从什么起点开始写”**。

如果起点只是一个空目录和一句“帮我做个页面”，你往往得到的是一份能截图、能演示、但不太能继续往下做的结果。它可能缺少清晰的目录职责，缺少状态层次，缺少测试入口，也缺少团队接手时最需要的工程约定。

所以，问题从来不只是“AI 会不会写 React”，而是“AI 生成出来的东西，是页面，还是项目”。

这就是 `react-monorepo-init` 这类 skill 值得被认真看待的原因。

## 你需要的，不是一个页面，而是一套专业起点

很多非研发角色卡住，并不是因为他们没有想法，也不是因为他们不会写 prompt，而是因为**项目初始化本身就是一门隐藏很深的前端工作**。

什么时候该用 Router？接口请求怎么组织？组件共享放在哪？本地状态和服务端状态怎么分？Tailwind、测试、目录结构、工程配置要不要一开始就带上？

这些问题，单独看都不复杂，合在一起却足以把一个原本想快速验证的需求拖进技术选型泥潭。

于是通常只会出现三种情况：

1. **停留在静态稿**  
   需求讲清楚了，界面也画出来了，但真正可交互、可试用的产品没有出现。

2. **得到一个一次性页面**  
   AI 很快拼出一个 demo，但当你想接 API、补路由、加测试、做第二个页面时，就发现结构开始塌。

3. **重新排队等前端资源**  
   团队最终还是回到传统模式，因为没人愿意在一个不成体系的初始产物上继续堆。

`react-monorepo-init` 的意义，就是把这段最容易卡住人的初始化工作，收敛成一套标准动作。

## `react-monorepo-init` 实际帮你搭了什么

它不是泛泛地说“给你一个 React 项目”，而是直接把一套现代前端基线搭出来：

- **React 19 + TypeScript**：现代主流基线，方便后续多人协作和 AI 持续补全。
- **Vite**：让开发体验保持轻快，不把项目第一步浪费在沉重构建配置上。
- **React Router**：从一开始就允许你把产品流程和页面路径组织起来。
- **TanStack Query**：服务端数据获取和缓存有规范入口，不容易一开始就写散。
- **Zustand**：本地状态管理保持轻量，适合 demo 和 MVP 的渐进复杂度。
- **Tailwind CSS v4**：让界面搭建速度足够快，同时保留后续演进空间。
- **Vitest / MSW / Playwright**：测试不是后补，而是基线里自带的质量入口。
- **pnpm monorepo 基线**：共享 UI、hooks、theme、配置包一开始就有落点。

它还支持几个常见增强项：`ci`、`storybook`、`pwa`、`i18n`。

更重要的是，它**有意不去解决所有问题**。后端、认证、部署平台、埋点方案，并不在默认范围内。因为如果在起步阶段把所有问题一起绑定，所谓“快速开始”很快就会变成“过度设计”。

## 跑完之后，你拿到的不是一个页面，而是一套像样的前端项目

这是最容易打动人的部分。

很多人第一次看到结果，不是被某个具体依赖打动，而是被项目结构本身说服。

```text
demo-web/
├─ apps/
│  └─ web/
├─ packages/
│  ├─ ui/
│  ├─ api-client/
│  ├─ shared/
│  ├─ hooks/
│  ├─ theme/
│  ├─ config-eslint/
│  ├─ config-typescript/
│  ├─ config-tailwind/
│  └─ config-vitest/
├─ .changeset/
├─ .husky/
├─ AGENTS.md
├─ DESIGN.md
├─ README.md
├─ CHANGELOG.md
└─ commitlint.config.js
```

**对非研发角色来说，这棵树意味着“我拿到的不是一个临时网页，而是一套真正能接着做的东西”。**

**对前端工程师来说，这意味着“这不是玩具脚手架，它已经把协作、质量和扩展位都预留出来了”。**

这也是为什么它会同时吸引非研发和前端同学。前者看到的是“终于可以自己把想法跑起来”，后者看到的是“终于不是又一坨只能推倒重来的代码”。

## 为什么前端同学也值得关心这件事

如果把这个 skill 理解成“非研发以后不需要前端了”，那就完全读错了。

它真正做的，是把前端工程里最重复、最容易被低估、却又经常拖慢项目启动的那一段标准化。

前端团队的时间，本来就不应该大量消耗在重复起目录、配基础工具链、定初版约定上。那些动作当然重要，但它们更适合被固化成模板和 skill，让团队把精力放在真正决定产品质量的部分：交互复杂度、状态边界、性能、可维护性、架构演进。

这背后的机制也很直接：**AI 对结构清晰、约定明确的代码库，会输出更稳定的结果。**

也就是说，标准化初始化不只是节省人力，它还会反过来提升后续 agent 协作的质量。

当然，代价也存在。`react-monorepo-init` 不是无限自由的搭建器。它更像是一套有明确品味的默认答案。你失去的是一部分初始化阶段的自由度，换来的是速度、一致性，以及更低的团队摩擦。

## 为什么最先受益的是产品、设计、运营、销售

**产品经理**最直接的变化，是可以把 PRD 变成可点击、可验证、可用户访谈的 MVP，而不是停在文档和流程图。

例如，一个产品同学想验证新的 onboarding 流程。以前他可能只能做一套静态稿，或者等前端抽时间。现在他至少可以和 AI 一起从标准基线出发，快速生成一个真正可走通的多页面流程。

**设计师**最直接的变化，是从“交付视觉稿”走向“交付接近真实行为的原型”。路由切换、状态反馈、接口模拟，这些以前往往断在设计交付之外，现在可以往前推进一段。

**运营同学**常常需要小工具、活动页、内部台账。真正麻烦的不是页面本身，而是每次都从零开局。标准基线让这类需求第一次有了可复用的起点。

**销售和解决方案团队**也一样。很多定制 demo 的价值窗口很短，等正式排期往往就错过时机。能不能快速起一个专业项目，直接决定了方案演示能不能抓住机会。

## 现在就可以怎么开始

如果你想直接试：

- npm: [@aicode-nexus/skills](https://www.npmjs.com/package/@aicode-nexus/skills?activeTab=readme)
- GitHub: [AICode-Nexus/skills](https://github.com/AICode-Nexus/skills)

先看当前已发布的 skills：

```bash
npx -y @aicode-nexus/skills list
```

安装 `react-monorepo-init`。如果你在 Codex 里使用，默认会装到 `~/.codex/skills`：

```bash
npx @aicode-nexus/skills install react-monorepo-init
```

如果你在 Claude Code 里使用，可以显式指定用户目录：

```bash
npx @aicode-nexus/skills install react-monorepo-init --dest ~/.claude/skills
```

如果你想把它先装进项目目录，方便团队共用、审阅或二次分发：

```bash
npx @aicode-nexus/skills install react-monorepo-init --dest ./tools/agent-skills
```

安装完成后，重新打开 agent 会话，就可以在真实工作流里调用它。

## 结语

AI coding 真正有用，不是因为它能一口气吐出很多代码，而是因为它能在一个正确的起点上持续工作。

把起点标准化，非研发角色才能第一次真正下场做产品验证；前端团队也才能把时间从重复搭建里解放出来。

如果你这周正好想做一个 demo、一个 MVP，或者一套内部工具，不妨从 `react-monorepo-init` 开始。  
先把场地搭对，再让 AI 开工。
