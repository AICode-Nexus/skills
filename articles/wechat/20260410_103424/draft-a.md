# Draft A - Deep Technical

## 标题方向

AI 能写页面不稀奇，能起一套专业 React 前端基线才重要

过去一年，很多人第一次感受到 AI coding 的力量，不是因为它能写复杂算法，而是因为它能快速把一句需求变成一个看得见的页面。

但真正做过项目的人都知道，**页面从来不是难点，项目才是难点**。一个看起来能跑的页面，和一个能继续开发、能测试、能多人协作、能逐步演进的前端项目，中间隔着的是初始化质量。

如果一个团队的起点只是“让 AI 先拼点页面出来”，最终常常会落到三个结果里：第一，产物只能演示，无法延续；第二，后续加路由、加状态、加接口、加测试时，结构会迅速失控；第三，前端工程师接手时，宁愿重建，也不愿修补。

这也是 `react-monorepo-init` 的价值所在。它解决的不是“怎么再快一点画一个页面”，而是“怎么给 AI 一个足够专业的 React 项目起点”。

## 它实际搭出来的，不是脚手架口号，而是工程基线

这个 skill 的核心不是抽象宣传，而是一次性把现代 React 项目最容易分散决策的部分固定下来。

- **React 19 + TypeScript**：保证主流类型化前端基线，避免从第一天开始就在“要不要上 TS”上反复。
- **Vite**：承担启动和构建基线，减少项目刚起步时的工具链摩擦。
- **React Router**：让多页面流转、业务路径、信息架构从一开始就是显式的。
- **TanStack Query**：把服务端状态管理纳入规范，避免接口请求散落在组件里。
- **Zustand**：处理本地状态时保持轻量，不把所有状态问题都塞进一种模式。
- **Tailwind CSS v4**：提供快速 UI 迭代基线，让 demo 和 MVP 阶段能低摩擦推进。
- **Vitest / MSW / Playwright**：把单元测试、接口模拟、端到端验证预埋进去，避免“以后再补测试”变成永远不补。
- **pnpm monorepo baseline**：从目录层面预留共享 UI、hooks、theme、config 包，让后续扩展不必拆家式重构。

可选增强也被收束为几个高频方向：`ci`、`storybook`、`pwa`、`i18n`。这其实是一个很重要的设计信号: **它是有边界的**。

它不在默认职责里帮你一口气决定后端、认证、部署平台、埋点方案。原因不是做不到，而是这些问题一旦混进初始化，就会把“标准起点”变成“早期过度设计”。

## 为什么初始化质量，会直接影响 AI 输出质量

AI 不是在真空里写代码。它总是在某个上下文里续写。

如果上下文是一堆散乱文件、模糊目录、缺失约定，那么 AI 产出的代码会天然更随机，风格更不稳定，边界更难维持。反过来，如果上下文已经有清晰的目录职责、测试入口、共享包位置、文档骨架和配置基线，AI 的补全和扩展会显著更稳。

这背后的机制很朴素：**模型不是只靠“理解需求”工作，它也靠“读取现有结构”工作。**

因此，初始化这一步不是一次性的体力活，而是后续所有 AI 协作质量的上游变量。

## 真正有说服力的，是跑完之后的目录

如果只听技术栈名称，很多人还是会把它理解成“又一个脚手架”。

但一旦你看到结果，感受会完全不同：

```text
demo-web/
├─ apps/
│  └─ web/
├─ packages/
│  ├─ ui/
│  ├─ api-client/
│  ├─ shared/
│  ├─ hooks/
│  ├─ theme/
│  ├─ config-eslint/
│  ├─ config-typescript/
│  ├─ config-tailwind/
│  └─ config-vitest/
├─ .changeset/
├─ .husky/
├─ AGENTS.md
├─ DESIGN.md
├─ README.md
├─ CHANGELOG.md
└─ commitlint.config.js
```

这棵树的意义不在于“文件很多”，而在于**职责已经被预先分层**。

对非研发角色来说，它传递的是确定感：这不是一页孤零零的 demo，而是一套像样的项目。对前端同学来说，它传递的是可信度：这不是一次性拼装代码，而是已经有共享包、工程配置、版本说明、团队约定和质量入口的基线。

## 前端工程师为什么也应该关心

这个 skill 不是来替代前端工程师的。真正更准确的说法是：它试图把前端工程师最不值得重复消耗的初始化劳动标准化。

前端团队经常把大量时间花在同类决策上：目录怎么切、配置怎么抽、测试怎么起、共享 UI 包什么时候拆、文档骨架要不要带上。这些决策当然重要，但它们又高度重复。

`react-monorepo-init` 的价值，在于把这些高频重复动作前置成一个稳定基线。这样团队可以把工程判断力用在真正有差异化的部分，而不是每次都从零布置场地。

它的代价也很明确：**更少自由度，换更高一致性**。如果你的团队本来就有成熟内部脚手架，这个 skill 未必替代它；但对于 AI-native 的实验项目、小团队、多角色协作场景，它能显著降低 bring-up 成本。

## 为什么最先受益的是非前端研发角色

产品经理最常见的痛点，是 PRD 已经足够清楚，但验证版本迟迟等不到。设计师的痛点，则是高保真稿很多，但交互、路由、状态切换并没有真实发生。运营和销售经常需要内部工具、活动页面、客户定制 demo，却很难为这类轻量需求争取稳定前端资源。

当一个 skill 能把项目起点搭好，这些角色就第一次拥有了“和 AI 一起往下推进”的地面，而不是停留在想法层。

他们当然不需要一夜之间变成前端工程师，但他们可以跨过最陡的第一段坡。

## 怎么开始

包地址：

- npm: https://www.npmjs.com/package/@aicode-nexus/skills?activeTab=readme
- GitHub: https://github.com/AICode-Nexus/skills

你可以先查看有哪些已发布 skills：

```bash
npx -y @aicode-nexus/skills list
```

安装 `react-monorepo-init`：

```bash
npx @aicode-nexus/skills install react-monorepo-init
```

装到 Claude Code 用户目录：

```bash
npx @aicode-nexus/skills install react-monorepo-init --dest ~/.claude/skills
```

装到项目目录，方便团队共享：

```bash
npx @aicode-nexus/skills install react-monorepo-init --dest ./tools/agent-skills
```

## 结尾

AI coding 真正进入产品化阶段之后，新的竞争力不会只是“谁能更快写出第一屏”，而是“谁能更快把一个想法放进正确的工程骨架里”。

`react-monorepo-init` 想做的，正是这件事。
